# Code Layout Rules

**Scope:** Code formatting: indentation, spacing, blank lines, line breaks.
**Core:** logical structure clarity, consistent indentation — always apply.
**Extended:** brace placement, line length, alignment details.
**Often used with:** comments.md, writing.md.
**Read Next If:** formatting code → comments.md; project style conventions → writing.md.

## Formatting Fundamentals

- [核心][必须] 布局的首要任务是准确展现代码的逻辑结构，而非追求视觉美观。 | anchor=CC2.CH31.S01
- [扩展][必须] 选择能更好展现代码逻辑结构的布局方案，优先于纯粹美观的方案。 | anchor=CC2.CH31.S01
- [扩展][必须] 布局应使人和计算机对程序有同样的理解，程序外观应与实际逻辑结构一致。 | anchor=CC2.CH31.S01
- [扩展][必须] 布局应准确表现代码的逻辑结构。 | anchor=CC2.CH31.S01
- [扩展][必须] 布局应始终如一地表现代码的逻辑结构。 | anchor=CC2.CH31.S01
- [扩展][必须] 布局应改善可读性。 | anchor=CC2.CH31.S01
- [扩展][必须] 布局应经得起修改——修改某行时不必连带修改其他行的代码。 | anchor=CC2.CH31.S01
- [扩展][必须] 前后一致地组织程序，比细节上选择哪种特定方法更重要。 | anchor=CC2.CH31.S01
- [扩展][禁止] 不要因为个人偏好或信仰争论而采用客观上更差的布局风格。 | anchor=CC2.CH31.S01
## Whitespace and Indentation

- [扩展][必须] 使用空白（空格、制表符、换行、空行）作为展现代码结构的主要手段。 | anchor=CC2.CH31.S02
- [扩展][必须] 将相关的语句成组放在一起。 | anchor=CC2.CH31.S02
- [扩展][必须] 用空行将不相关的语句分隔开。 | anchor=CC2.CH31.S02
- [扩展][必须] 用空行将相关语句各自划分成段落、分开各子程序、突出注释部分。 | anchor=CC2.CH31.S02
- [核心][必须] 用缩进形式显示程序的逻辑结构——逻辑上从属于某语句的语句都应缩进。 | anchor=CC2.CH31.S02
- [扩展][必须] 缩进 2 到 4 个空格。（研究表明缩进二到四个空格最得当，受试者对程序的理解分数比对毫无缩进的程序高出 20% 到 30%）| anchor=CC2.CH31.S02
- [核心][禁止] 不要使用无缩进的代码。 | anchor=CC2.CH31.S02
- [扩展][禁止] 不要缩进 6 个或更多空格。 | anchor=CC2.CH31.S02
- [扩展][必须] 对包含两个以上项的表达式使用括号加以澄清。 | anchor=CC2.CH31.S02
## Control Structure Layout

- [语言特定][必须] 使用 Visual Basic 时采用纯块缩进结构（即控制结构自有结束符，块内语句一致缩进）。 | anchor=CC2.CH31.S03
- [语言特定][必须] 使用 Java 时采用纯块缩进结构。 | anchor=CC2.CH31.S03
- [语言特定][必须] 使用 C++ 时，选择模仿纯块结构或将 begin-end 视为块边界的风格之一，并在团队中统一使用。（两种风格在可理解性方面并无统计上的显著差别 (Hansen and Yim 1987)，关键是一致使用）| anchor=CC2.CH31.S03
- [扩展][禁止] 如果所用语言不支持纯块结构，不要在控制语句后随意放置 begin/end 而不遵循统一缩进规范。 | anchor=CC2.CH31.S03
- [扩展][必须] 在模仿纯块结构时，应将 begin（花括号）放在控制结构的同一行末尾，将 end 放在相应缩进级别。 | anchor=CC2.CH31.S03
- [扩展][禁止] 不要使用"行尾布局"风格作为主要布局策略。 | anchor=CC2.CH31.S03
- [扩展][禁止] 不要在 case 语句中使用行尾布局（将 case 代码缩进到 case 标签右侧）。 | anchor=CC2.CH31.S03
- [扩展][禁止] 不要使用未缩进的 begin-end 对（即 begin-end 与控制结构左对齐，其间语句在 begin 下缩进）。 | anchor=CC2.CH31.S04
- [扩展][禁止] 不要对 begin-end 对做两次缩进（即 begin 和 end 缩进一次，其间语句再缩进一次）。 | anchor=CC2.CH31.S04
- [扩展][必须] 对未用 begin-end 对界定的逻辑代码块，使用空行将它们分隔开，如同英语中的段落。 | anchor=CC2.CH31.S04
- [扩展][必须] 对控制结构后的单语句代码块，其格式要保持前后统一。 | anchor=CC2.CH31.S04
- [扩展][必须] 对于复杂的表达式，将条件的各个部分分隔放在各自的几行上。 | anchor=CC2.CH31.S04
- [扩展][禁止] 不要把复杂的表达式全部挤在一行里。 | anchor=CC2.CH31.S04
- [历史][禁止] 尽量不要使用 goto。 | anchor=CC2.CH31.S04
- [历史][必须] 如果必须使用 goto，goto 标号应完全大写并左对齐于行首。 | anchor=CC2.CH31.S04
- [历史][必须] 如果必须使用 goto，包含 goto 的语句应单独为一行。 | anchor=CC2.CH31.S04
- [历史][必须] 如果必须使用 goto，goto 语句转到的标号应单独为一行，上下也为空行。 | anchor=CC2.CH31.S04
- [扩展][禁止] 不要为 case 语句创造行尾布局的例外——case 语句应按标准缩进增量缩进。 | anchor=CC2.CH31.S04
## Expressions and Statements

- [扩展][必须] 在逻辑表达式中使用空格以提高可读性。 | anchor=CC2.CH31.S05
- [扩展][必须] 在数组引用中使用空格以提高可读性——在数组下标前后加上空格。 | anchor=CC2.CH31.S05
- [扩展][必须] 在子程序参数中使用空格以提高参数的可读性。 | anchor=CC2.CH31.S05
- [扩展][必须] 当语句需要分成多行时，使续行明显——放在第一行中的部分应清楚地表明它仅是语句的一部分（例如以运算符结尾）。 | anchor=CC2.CH31.S05
- [扩展][必须] 将后续行的运算符置于行首，以标明运算的结构并使续行可见。 | anchor=CC2.CH31.S05
- [扩展][必须] 断句时将紧密关联的元素放在一起（如数组下标、子程序参数等不应被拆散到不同行）。 | anchor=CC2.CH31.S05
- [扩展][必须] 将子程序调用的后续行按标准缩进量缩进（与循环/条件语句的缩进量相同）。 | anchor=CC2.CH31.S05
- [扩展][必须] 将控制语句（for/while/if）的后续行缩进以标准的空格数，与循环体或 if 后的语句一样缩进。 | anchor=CC2.CH31.S05
- [扩展][禁止] 不要将赋值语句按等号对齐（即行尾布局式的等号对齐）。 | anchor=CC2.CH31.S05
- [扩展][必须] 对赋值语句的后续行按标准缩进量缩进，与其他语句同等对待。 | anchor=CC2.CH31.S05
- [核心][必须] 每行仅写一条语句。 | anchor=CC2.CH31.S05
- [扩展][禁止] 不要在一行里放多条语句。 | anchor=CC2.CH31.S05
- [语言特定][禁止] 在 C++ 中，不要让一行里有多个操作（副作用），如将 ++ 操作符与其他操作混在一行。 | anchor=CC2.CH31.S05
- [扩展][必须] 将包含副作用的语句单独放在一行，与主操作分离。 | anchor=CC2.CH31.S05
- [扩展][必须] 每行只声明一个数据。 | anchor=CC2.CH31.S05
- [扩展][禁止] 不要将多个变量的声明堆在同一行或同一组中。 | anchor=CC2.CH31.S05
- [语言特定][必须] 在 C++ 中，声明指针变量时要么每行只声明一个变量，要么使用指针类型别名。 | anchor=CC2.CH31.S05
## Comment Layout

- [扩展][必须] 注释的缩进要与相应代码的缩进一致。 | anchor=CC2.CH31.S06
- [扩展][禁止] 不要让注释扰乱代码的缩进外观。 | anchor=CC2.CH31.S06
- [扩展][必须] 每组注释之前用至少一个空行分开，便于读者扫视。 | anchor=CC2.CH31.S06
## Subroutine and Class Layout

- [扩展][必须] 用空行分隔子程序的各部分——在子程序的头部、数据和常量声明（如果有）以及子程序体之间插入空行。 | anchor=CC2.CH31.S07
- [扩展][必须] 将子程序参数按标准缩进（而非行尾对齐风格）。 | anchor=CC2.CH31.S07
- [扩展][禁止] 不要使用行尾布局来对齐子程序参数。 | anchor=CC2.CH31.S07
- [扩展][必须] 子程序参数列表的续行应按标准缩进量缩进，每个参数占一行。 | anchor=CC2.CH31.S07
- [扩展][必须] 类接口的成员按以下顺序排布： 1）说明类及其用法的头部注释 2）构造函数与析构函数 3）public 子程序 4）protected 子程序 5）private 子程序和数据成员。 | anchor=CC2.CH31.S08
- [扩展][必须] 类实现的成员按以下顺序排布： 1）描述类所在文件内容的头部注释 2）类数据 3）public 子程序 4）protected 子程序 5）private 子程序。 | anchor=CC2.CH31.S08
- [扩展][必须] 如果一个文件包含多个类，要清楚地标出每一个类，相关子程序应分组放到类中。 | anchor=CC2.CH31.S08
- [扩展][必须] 用空行将类与相邻类分开，使其井井有条。 | anchor=CC2.CH31.S08
- [扩展][禁止] 不要在类里过度注释——避免对每个子程序都用星号行和多余标记。 | anchor=CC2.CH31.S08
- [扩展][禁止] 避免将两行星号或连字号放在一块。 | anchor=CC2.CH31.S08
- [扩展][必须] 一个文件应只有一个类。 | anchor=CC2.CH31.S08
- [扩展][必须] 文件的命名应与类名有关（如类名为 customerAccount，文件名应为 customerAccount.cpp 和 CustomerAccount.h）。 | anchor=CC2.CH31.S08
- [扩展][必须] 在文件中用至少两个空行将各子程序分开。 | anchor=CC2.CH31.S08
- [语言特定][必须] 仔细组织源文件中内容的次序。C++ 源文件的典型顺序为：1）文件描述性注释 2）#include 行 3）多类使用的常量定义 4）多类使用的枚举 5）宏函数定义 6）多类使用的类型定义 7）导入的全局变量和函数 8）导出的全局变量和函数 9）本文件私用的变量和函数 10）各个类。 | anchor=CC2.CH31.S08
- [扩展][必须] 布局方案应准确表现、始终如一表现代码的逻辑结构。（见 31.1 节格式化基本原理及良好布局的三个目标）| anchor=CC2.CH31.S01
- [扩展][必须] 布局方案应改善可读性且经得起修改。（见 31.1 节良好布局的目标）| anchor=CC2.CH31.S01
- [扩展][禁止] 不要在 if/while/for 等控制结构后忘记添加 begin-end 对。 | anchor=CC2.CH31.S04
- [扩展][禁止] 不要在需要时省略括号——用括号比不用好。 | anchor=CC2.CH31.S02
- [核心][必须] 在整个项目中统一布局风格，不能持久布局规范只会损害可读性。 | anchor=CC2.CH31.S03
